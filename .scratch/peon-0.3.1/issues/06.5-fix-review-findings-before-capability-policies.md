## Host-ownership boundary (spec §"Controller and Host Ownership")

Textual still persists config directly in ~20 call sites. Ticket 06 added
controller-routed outcome handlers alongside old direct-persistence paths
instead of replacing them. The spec says: "Textual must not persist
config/messages, select provider policy, execute tools, or mutate
session/controller state outside intent dispatch."

- [ ] Remove direct `config_store.save()`, `.delete()`, `.load_all()`,
  `save_ui_config()`, and `update_saved_provider()` calls from Textual widget
  methods that duplicate controller-owned flows (provider setup, model
  selection, logout, settings persistence).
- [ ] Textual provider/model/settings/logout widget actions dispatch intents
  to the controller and react to outcomes only — no direct store access for
  those flows.
- [ ] `config_store` may remain on `TextualPeonApp` for read-only presentation
  needs (e.g., listing providers for display) and for UI-config preferences
  that are presentation-only state (theme, padding), but must not mutate
  provider/session/capability state.
- [ ] `LogoutSuccessOutcome` handler must not re-derive active provider by
  querying the store; use the controller-provided `active_provider_name` or
  accept a richer outcome.

## Controller duck-typing cleanup
